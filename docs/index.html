Common Integration Scenarios

The kit is designed for customer teams that need a stable contract for exchanging wafer context, control commands, status, result summaries, and error state with a Virex.NET-compatible service.

Send wafer context from a host or MES

Use REST or TCP to send WaferInfo before starting an inspection cycle. The public fields are lotId, waferId, recipeId, slot, foupId, and chamberId.

Control a simulated inspection cycle

Use REST control endpoints or TCP command frames to initialize, start, stop, and terminate a cycle while validating client-side error handling and state transitions.

Subscribe to operational events

Use TCP when the integration expects a direct socket connection. Use MQTT when another system needs outbound event distribution through a broker.

Query summary results

Use REST to query result summaries by lotId, waferId, and recipeId. Result payloads are summary-only and intentionally exclude large inspection data.

Choose an Interface

REST, TCP, and MQTT are all supported, but they are intended for different integration styles. Choose the narrowest interface that matches the customer's system boundary.

Interface Best fit Direction Typical use
REST Request/response commands and queries Client to service Read status, send wafer info, initialize, start, stop, terminate, and query results.
TCP / NDJSON Direct socket integration Bidirectional Send wafer info or command frames and receive status, wafer info, result, and error events.
MQTT Brokered event subscription Outbound events only Subscribe to status, wafer info, result, and error topics. MQTT is not used for command/control.

C# SDK Quick Start

Use the SDK when the integrating application is written in C#. It wraps the public communication APIs with a small surface and targets netstandard2.0 for broad application compatibility. 

using Virex.NET.Client;
using Virex.NET.Contracts;

using var client = new VirexClient(new VirexClientOptions
{
    RestBaseUrl = "http://127.0.0.1:5088",
    TcpHost = "127.0.0.1",
    TcpPort = 5089,
    MqttHost = "127.0.0.1",
    MqttPort = 1883,
    MqttTopic = "Virex.NET",
    TimeoutMs = 5000,
});

var status = await client.GetStatusAsync();

await client.SetWaferInfoAsync(new WaferInfo
{
    LotId = "LOT-001",
    WaferId = "W01",
    RecipeId = "RCP-A",
    Slot = "1",
    FoupId = "FOUP-A",
    ChamberId = "CH-1",
});

await client.InitializeAsync();
await client.StartAsync();

var results = await client.QueryResultsAsync(lotId: "LOT-001");

REST convenience methods

VirexClient exposes GetStatusAsync, SetWaferInfoAsync, InitializeAsync, TerminateAsync, StartAsync, StopAsync, and QueryResultsAsync.

Error handling

Non-success HTTP responses throw VirexClientException, including the numeric status code and response body so customer applications can log or display the service response.

Event Subscription

Event clients expose a common EventReceived pattern. The parsed event contains the original raw JSON plus the typed payload for supported event types. 

using var cts = new CancellationTokenSource();

client.MqttEvents.EventReceived += (_, ev) =>
{
    if (ev.Status is not null)
        Console.WriteLine($"State: {ev.Status.ProcessState}");

    if (ev.Result is not null)
        Console.WriteLine($"Result: {ev.Result.ResultId} {ev.Result.OverallResult}");
};

await client.MqttEvents.RunAsync(cts.Token);
MQTT topic behavior MQTT messages are published under the configured base topic. The default child topics are status, wafer-info, result, and error. The MQTT client attaches the event type from the topic when the payload does not include a type field.

Data Contract Summary

WaferInfo

Identifies the wafer and surrounding production context. Required validation for TCP wafer info frames includes non-empty lotId, waferId, slot, foupId, and chamberId.

StatusDto

Reports whether the service is initialized, the current process state, and the active recipe label. Public process states are ready, capturing, inspecting, and saving.

ResultSummaryDto

Provides fixed-size result summaries such as result id, timestamp, wafer context, overall result, defect count, die count, and relative output paths.

ErrorStatusDto

Reports the current error message together with initialized state, process state, and recipe so clients can show actionable operational context.

Result payload boundary Result responses and result events are summary-only. They do not include defect lists, die lists, crop lists, image binaries, or private inspection details.

Simulator Workflow

Use the WPF simulator to validate integration code before connecting to a Virex.NET-compatible service. The simulator exposes the same public REST, TCP, and MQTT communication surfaces.

  1. Build and test the repository

    dotnet test Virex.NET.Integration.slnx

  2. Start the simulator

    dotnet run --project src\Virex.NET.Simulator.WPF\Virex.NET.Simulator.WPF.csproj

  3. Run the C# SDK sample

    dotnet run --project samples\csharp-sdk\CSharpSdkSample.csproj

  4. Validate customer integration behavior

    Confirm that the customer application can send wafer context, control the simulated cycle, receive status/result/error events, and query result summaries using the chosen interface.

Troubleshooting

REST request fails

Verify RestBaseUrl, firewall rules, simulator status, and the HTTP response body carried by VirexClientException.

TCP client does not receive events

Confirm the TCP host and port, ensure frames are UTF-8 NDJSON, and keep the event loop running with a live cancellation token.

MQTT events are missing

Confirm broker connectivity, port 1883, base topic Virex.NET, and subscription to the configured topic tree.

Result query is empty

Verify the simulator has emitted a result and that lotId, waferId, and recipeId filters match the submitted wafer context.
